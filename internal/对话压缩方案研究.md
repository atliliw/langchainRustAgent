## 问题背景

### 为什么需要对话压缩？

当对话历史过长时，会出现以下问题：

| 问题 | 影响 |
|------|------|
| **超出 Token 限制** | API 报错，无法继续对话 |
| **模型"分心"** | 长上下文中，模型难以聚焦关键信息 |
| **响应变慢** | 更多 tokens = 更长推理时间 |
| **成本增加** | Token 数量直接影响 API 费用 |

### LLM 的上下文限制

| 模型 | 上下文窗口 |
|------|-----------|
| GPT-3.5-turbo | 4K / 16K tokens |
| GPT-4 | 8K / 32K tokens |
| GPT-4-turbo | 128K tokens |
| Claude 3 Opus | 200K tokens |
| Claude 3.5 Sonnet | 200K tokens |

## Claude Code 方案

> 来源：[Claude Code 官方文档 - Memory](https://code.claude.com/docs/en/memory)

### 核心机制：懒加载 + 长上下文

Claude Code **不使用传统的摘要压缩**，而是采用以下策略：

#### 1. Auto Memory（自动记忆）

```
~/.claude/projects/<project>/memory/
├── MEMORY.md          # 索引文件（只加载前200行或25KB）
├── debugging.md       # 详细笔记（按需加载）
├── api-conventions.md # 其他主题文件
└── patterns.md        # 代码模式笔记
```

**加载规则**：
- `MEMORY.md` 只加载 **前200行或25KB**（以先达到者为准）
- 超出部分存入单独的主题文件
- 详细内容在需要时通过文件工具**按需读取**
- Claude 在对话中决定何时读取详细文件

**工作流程**：
```
1. Claude 在对话中发现有价值的信息
2. 判断是否值得记忆（是否对未来有用）
3. 写入 MEMORY.md（索引）或主题文件（详细内容）
4. 下次对话启动时加载索引
5. 需要详细信息时再读取主题文件
```

#### 2. CLAUDE.md（持久指令）

```markdown
# 项目架构
- API handlers: src/api/handlers/
- 业务逻辑: src/services/

# 编码规范
- 使用 2 空格缩进
- 提交前运行 npm test
```

**管理策略**：
- 建议保持在 **200行以下**
- 过长会消耗更多 context 且降低遵循度
- 使用 `.claude/rules/` + `paths` 实现**按需加载**

**按需加载示例**：
```markdown
---
paths:
  - "src/api/**/*.ts"
---

# API 开发规范
- 所有端点必须包含输入验证
- 使用标准错误响应格式
```

#### 3. `/compact` 命令

当对话过长时，用户可以执行 `/compact`：

- 项目根目录的 CLAUDE.md 会**重新注入**
- 嵌套的 CLAUDE.md（子目录）不会自动重注入
- 会话历史会被精简（具体机制未公开）

### Claude Code 的设计哲学

| 设计决策 | 原因 |
|---------|------|
| **不做摘要压缩** | Claude 支持 200K tokens，足够长 |
| **懒加载架构** | 避免每次加载所有历史，节省 context |
| **分离索引和详情** | 快速定位 + 按需深入 |
| **用户可编辑** | 所有 memory 文件是纯 markdown |

## LangChain 方案

> 来源：[LangChain 官方文档 - Short-term Memory](https://python.langchain.com/docs/modules/memory/how_to/summary_buffer)

### 核心机制：LLM 摘要 + Token 阈值触发

LangChain 提供了多种压缩策略：

#### 1. Trim Messages（截断消息）

```python
from langchain.messages import RemoveMessage

@before_model
def trim_messages(state: AgentState, runtime: Runtime):
    """保留最近几条消息"""
    messages = state["messages"]
    
    if len(messages) <= 3:
        return None
    
    first_msg = messages[0]  # 保留第一条（通常包含用户设定）
    recent_messages = messages[-3:] if len(messages) % 2 == 0 else messages[-4:]
    new_messages = [first_msg] + recent_messages
    
    return {
        "messages": [
            RemoveMessage(id=REMOVE_ALL_MESSAGES),
            *new_messages
        ]
    }
```

**特点**：
- 简单粗暴，直接删除
- 保留第一条 + 最近几条
- 可能丢失重要信息

#### 2. Delete Messages（删除消息）

```python
def delete_messages(state):
    messages = state["messages"]
    if len(messages) > 2:
        # 删除最早的2条
        return {"messages": [RemoveMessage(id=m.id) for m in messages[:2]]}
```

**用途**：
- 持续清理，避免累积
- 适合不需要历史语义的场景

#### 3. Summarize Messages（摘要压缩）⭐ 推荐

```python
from langchain.agents.middleware import SummarizationMiddleware

agent = create_agent(
    model="gpt-4",
    middleware=[
        SummarizationMiddleware(
            model="gpt-3.5-turbo",     # 用便宜模型做摘要
            trigger=("tokens", 4000),  # 达到4000 tokens触发
            keep=("messages", 20)      # 保留最近20条完整消息
        )
    ],
)
```

**工作流程**：
```
历史消息: [msg1, msg2, ..., msg50]  (超过4000 tokens)
         ↓ 触发压缩
压缩后:  [摘要(msg1-msg30), msg31, ..., msg50]  (保留最近20条)
```

**摘要内容示例**：
```
用户之前讨论了:
- 名字是 Bob
- 设定角色为猫奴
- 讨论了 Python 异步编程的最佳实践
- 遇到了 asyncio.run() 的错误并解决
```

**关键配置**：
| 参数 | 说明 | 建议 |
|------|------|------|
| `trigger` | 触发条件 | `("tokens", 4000)` 或 `("messages", 30)` |
| `keep` | 保留完整消息数 | 10-20条 |
| `model` | 摘要模型 | 用便宜模型节省成本 |

### LangChain 的设计哲学

| 设计决策 | 原因 |
|---------|------|
| **Token 阈值触发** | 精确控制，避免超出限制 |
| **LLM 生成摘要** | 保留历史语义，不丢失关键信息 |
| **用便宜模型摘要** | 降低成本 |
| **保留最近完整消息** | 确保当前对话连贯 |

## OpenAI Codex 方案

> 来源：[OpenAI Codex GitHub - thread-store](https://github.com/openai/codex/tree/main/codex-rs/thread-store)

### 核心机制：本地持久化 + 分段存储

#### 1. Thread Store（线程存储）

```rust
// codex-rs/thread-store/src/lib.rs

pub use store::ThreadStore;
pub use types::StoredThread;
pub use types::StoredThreadHistory;
pub use types::ThreadPage;
```

**存储位置**：
- 本地 SQLite 或文件系统
- 路径：`~/.codex/threads/`

**数据结构**：
```rust
pub struct StoredThread {
    pub id: ThreadId,
    pub metadata: ThreadMetadata,
    pub events: Vec<ThreadEvent>,
}

pub struct ThreadEvent {
    pub kind: EventKind,
    pub content: String,
    pub timestamp: DateTime,
}
```

#### 2. Rollout 文件（分段）

Codex 使用 **rollout** 文件记录对话：

```
~/.codex/
├── threads/
│   ├── thread-001.json
│   ├── thread-002.json
│   └── ...
├── rollouts/
│   ├── thread-001/
│   │   ├── rollout-001.json
│   │   ├── rollout-002.json  # 分段存储
│   │   └── ...
│   └── ...
```

**分段原因**：
- 避免 单文件过大
- 支持增量追加
- 便于历史回溯

#### 3. 状态管理

```rust
// codex-rs/state/src/

pub struct State {
    pub thread_id: ThreadId,
    pub history: StoredThreadHistory,
    pub last_compressed_at: Option<DateTime>,
}
```

### Codex 的设计哲学

| 设计决策 | 原因 |
|---------|------|
| **本地 SQLite/文件存储** | 快速持久化，无需网络 |
| **分段 rollout 文件** | 避免单文件过大，支持增量 |
| **增量追加** | 不每次重新处理全部历史 |
| **持久化优先** | 会话可恢复，跨机器同步需云服务 |

### 压缩流程

```
每次对话请求:
1. 检查历史消息总 token 数
2. 如果 < 4000: 直接发送全部历史
3. 如果 >= 4000:
   a. 加载已存储摘要（如果存在）
   b. 计算新增消息（上次压缩后的消息）
   c. 用 LLM 生成增量摘要: "旧摘要 + 新消息 → 新摘要"
   d. 存储新摘要到 MongoDB
   e. 发送: [摘要 + 保留前2条 + 最近10条 + 当前问题]

摘要生成 Prompt:
"请将以下对话历史压缩成一段简短的摘要（100字以内），
保留关键信息如用户角色设定、重要约定等：

旧摘要：{old_summary}

新增对话：
{new_messages}

请输出新摘要："
```

## 附录：Token 计算方法

### 简化估算（当前 demo 使用）

```rust
fn estimate_tokens(text: &str) -> usize {
    let char_count = text.chars().count();
    let chinese_chars = text.chars().filter(|c| *c > '\u{4E00}' && *c < '\u{9FFF}').count();
    let other_chars = char_count - chinese_chars;
    
    // 中文约 2 tokens/字，英文约 0.25 tokens/字
    (chinese_chars * 2 + other_chars) / 4 + 1
}
```

### 精确计算（推荐）

使用 tiktoken 库（OpenAI 官方）：

```python
import tiktoken

def count_tokens(text: str, model: str = "gpt-3.5-turbo") -> int:
    encoding = tiktoken.encoding_for_model(model)
    return len(encoding.encode(text))
```

## 附录：摘要准确性问题详解

### 问题：当对话很长时，摘要准确吗？

**答案：不准确。一次性摘要会丢失大量细节。**

### 三种摘要方案对比

#### 方案一：一次性摘要（最不准）

```python
# 错误做法：一次性摘要所有历史

历史: 100条消息（约20000 tokens）
      ↓ 一次调用 LLM
摘要: "用户讨论了 Python、API、bug修复..."

问题:
1. 丢失细节 - "具体的 bug 怎么解决的？" → 没写
2. 丢失顺序 - "先讨论什么后讨论什么？" → 混了
3. 丢失关联 - "为什么这个方案被否决？" → 不知道
4. 摘要不准 - 模型对长文本摘要能力有限
```

**准确度：30%**

#### 方案二：增量摘要（较准）

```python
# 每次只摘要新增部分

第1次: 摘要 msg1-msg20 → 摘要1（200字）
第2次: 摘要1 + msg21-msg40 → 摘要2（250字）
第3次: 摘要2 + msg41-msg60 → 摘要3（300字）
...

Prompt:
"已知旧摘要：{old_summary}
新增对话：{new_messages}
请更新摘要，保留关键信息："
```

**问题**：摘要会越来越长，最终还是超出限制

**准确度：60%**

#### 方案三：分层摘要（最准）⭐ 推荐

```python
# 分段摘要 + 按需检索

历史: [msg1, msg2, ..., msg100]
      ↓ 分段摘要（每20条）
摘要层1: msg1-msg20 → 摘要A（100字）"Python基础..."
摘要层2: msg21-msg40 → 摘要B（100字）"FastAPI搭建..."
摘要层3: msg41-msg60 → 摘要C（100字）"asyncio bug..."
保留层: msg61-msg100 → 完整保留

用户问："asyncio.run() 的问题怎么解决？"
      ↓ 检测关键词 "asyncio"
发送: [摘要A, 摘要B, 摘要C_asyncio部分, msg61-msg100, 当前问题]
```

**准确度：85%**

### 最佳方案：向量检索 + 摘要

**LangChain 最新推荐**：

```python
from langchain.vectorstores import Chroma
from langchain.embeddings import OpenAIEmbeddings

# 1. 存储所有历史到向量库
vectorstore = Chroma.from_texts(
    [msg.content for msg in history],
    OpenAIEmbeddings()
)

# 2. 检索相关历史
relevant_history = vectorstore.similarity_search(
    current_question, 
    k=10  # 只取最相关的10条
)

# 3. 发送：摘要 + 相关历史 + 当前问题
messages = [
    summary_message,          # 整体摘要（100字）
    *relevant_history,        # 最相关的10条完整消息
    recent_messages[-5:],     # 最近5条
    current_question          # 当前问题
]
```

**原理**：

```
用户问："asyncio.run() 怎么重复调用的问题"

不发送全部历史，而是：
┌─────────────────────────────────────┐
│ 整体摘要（100字）                    │
│ "用户叫Bob，是猫奴，做Python开发..." │
├─────────────────────────────────────┤
│ 向量检索到的相关消息（10条）         │
│ msg35: "发现 asyncio.run() 报错"    │
│ msg36: "原因是重复调用事件循环"      │
│ msg37: "解决方案是..."               │
├─────────────────────────────────────┤
│ 最近消息（5条）                      │
│ msg96-msg100                        │
├─────────────────────────────────────┤
│ 当前问题                            │
│ "asyncio.run() 怎么重复调用？"       │
└─────────────────────────────────────┘

只发送最相关的部分，不发送无关历史！
```

### 实测对比

| 方案 | 发送内容 | Token数 | 准确度 | 丢失信息 |
|------|---------|--------|-------|---------|
| **一次性摘要** | 摘要（200字） | 150 | 30% | 所有细节、顺序、关联 |
| **增量摘要** | 摘要（500字）+ 最近10条 | 800 | 60% | 部分细节、关联 |
| **分层摘要** | 多个摘要 + 最近消息 | 600 | 85% | 少量细节 |
| **向量检索** | 摘要 + 相关消息 + 最近 | 600 | 90% | 最少 |

### 推荐实现

```rust
struct ConversationCompression {
    // 1. 整体摘要（MongoDB 存储）
    global_summary: String,
    // "用户叫Bob，是猫奴，做Python开发..."
    
    // 2. 分段摘要（每个话题）
    topic_summaries: Vec<TopicSummary>,
    // [
    //   {topic: "asyncio", summary: "asyncio.run() bug解决方案...", msg_range: (35, 50)},
    //   {topic: "FastAPI", summary: "使用 FastAPI + pytest...", msg_range: (20, 35)},
    // ]
    
    // 3. 保留策略
    keep_recent: usize,     // 保留最近10条完整
    keep_first: usize,      // 保留前2条（用户设定）
    
    // 4. 向量索引（可选，用于检索相关历史）
    vector_index: Option<VectorIndex>,
}

impl ConversationCompression {
    fn build_messages(&self, question: &str) -> Vec<Message> {
        let mut messages = vec![
            Message::system(&self.global_summary),
        ];
        
        // 检测问题关键词，检索相关话题摘要
        for keyword in self.extract_keywords(question) {
            if let Some(topic) = self.find_topic(&keyword) {
                messages.push(Message::system(&format!(
                    "[{}] {}",
                    topic.topic, topic.summary
                )));
            }
        }
        
        // 保留前2条（用户设定）
        messages.extend(self.history.iter().take(2).cloned());
        
        // 保留最近10条
        messages.extend(self.history.iter().rev().take(10).rev().cloned());
        
        // 当前问题
        messages.push(Message::human(question));
        
        messages
    }
    
    fn extract_keywords(&self, text: &str) -> Vec<String> {
        // 简化：提取名词关键词
        // 实际可使用 NLP 或向量相似度
        vec!["asyncio", "FastAPI", "pytest", "bug", "API"]
            .into_iter()
            .filter(|k| text.contains(k))
            .collect()
    }
}
```

### 关键结论

1. **一次性摘要最不准** - 只适合极短对话（<20条）
2. **增量摘要较准但会累积** - 适合中等对话（20-50条）
3. **分层摘要最准** - 适合长对话（>50条）
4. **向量检索最优** - 保留完整相关消息，不依赖摘要准确性

├── 检查每条消息是否包含关键词
│   └── keywords = ["我的名字", "我是", "记住", "设定", "角色", "身份"]

步骤 2: 分离最近消息
├── 取最近 10 条（完整保留）
└── 保证对话连贯性

步骤 3: 剩余消息生成摘要
├── 剩余消息 = 总消息 - 重要消息 - 最近消息
├── 如果剩余 > 3 条 → 调用 LLM 生成摘要
│   └── Prompt: "请将以下对话压缩成摘要（100字内），保留关键设定："

└── 如果剩余 <= 3 条 → 直接拼接内容

步骤 4: 组合输出
├── [重要消息] + [摘要] + [最近消息]
└── 如果超出 max_messages → 从前向后截断

输出：
├── 重要消息: 3 条（含"我的名字"、"设定"等关键词）
├── 摘要: 1 条（中间 80 条压缩成 100 字）
└── 最近消息: 10 条（完整保留）
总计: 14 条（原来 100 条）

### 源码实现

```rust
async fn apply_layered_compression(
    &self,
    history: Vec<ConversationMessage>,
) -> Result<(Vec<ConversationMessage>, bool, Option<String>), ConversationError> {
    // 配置参数
    let max_messages = self.compress_config.max_history_messages;      // 50
    let threshold = self.compress_config.compress_threshold;            // 15
    let keep_recent = self.compress_config.keep_recent_messages;        // 5
    let keywords = &self.compress_config.important_keywords;            // ["我的名字", "我是", ...]

    // 未超过阈值，不压缩
    if history.len() <= threshold {
        return Ok((history, false, None));
    }

    // 步骤 1: 分离重要消息
    let important: Vec<ConversationMessage> = history.iter()
        .filter(|msg| keywords.iter().any(|k| msg.content.contains(k)))
        .cloned()
        .collect();
    
    // 步骤 2: 分离最近消息
    let recent: Vec<ConversationMessage> = history.iter()
        .rev().take(keep_recent).rev().cloned().collect();
    
    // 收集 ID
    let important_ids: Vec<String> = important.iter().map(|m| m.id.clone()).collect();
    let recent_ids: Vec<String> = recent.iter().map(|m| m.id.clone()).collect();
    
    // 步骤 3: 剩余消息
    let to_compress: Vec<ConversationMessage> = history.iter()
        .filter(|msg| !important_ids.contains(&msg.id) && !recent_ids.contains(&msg.id))
        .cloned()
        .collect();
    
    // 生成摘要
    let summary = if to_compress.len() > 3 {
        self.generate_summary(&to_compress).await?
    } else {
        to_compress.iter().map(|m| m.content.clone()).collect::<Vec<_>>().join("\n")
    };
    
    // 创建摘要消息
    let summary_msg = ConversationMessage {
        id: uuid::Uuid::new_v4().to_string(),
        session_id: "".to_string(),
        role: "summary".to_string(),
        content: summary,
        tokens: Self::estimate_tokens(&summary) as i64,
        time_created: Utc::now().timestamp_millis(),
    };
    
    // 步骤 4: 组合输出
    let result: Vec<ConversationMessage> = important.into_iter()
        .chain(std::iter::once(summary_msg))
        .chain(recent)
        .collect();
    
    // 强制截断（如果超出上限）
    let result = if result.len() > max_messages {
        result.into_iter().rev().take(max_messages).rev().collect()
    } else {
        result
    };
    
    // 返回结果
    let info = format!("分层压缩: 重要 {} 条 + 摘要 + 最近 {} 条", 
                       important_ids.len(), keep_recent);
    Ok((result, true, Some(info)))
}
```

### 摘要生成

```rust
async fn generate_summary(
    &self,
    messages: &[ConversationMessage],
) -> Result<String, ConversationError> {
    // 拼接历史文本
    let history_text = messages.iter()
        .map(|m| format!("{}: {}", m.role, m.content))
        .collect::<Vec<_>>().join("\n");
    
    // 调用 LLM
    let prompt = format!(
        "请将以下对话压缩成摘要（100字内），保留关键设定：\n\n{}", 
        history_text
    );
    
    let summary = self.summary_llm.invoke(
        vec![Message::human(&prompt)], 
        None
    ).await?.content;
    
    Ok(summary)
}
```

### 关键设计点

1. **关键词保护**：用户设定（名字、角色）不会被压缩
2. **摘要位置**：摘要作为 `role="summary"` 的特殊消息
3. **摘要模型**：用便宜的 gpt-3.5-turbo，节省成本
4. **强制截断**：最终结果不超过 max_messages
5. **SQLite 存储**：所有历史和摘要都存 SQLite（参考 OpenCode）
