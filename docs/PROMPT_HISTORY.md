### 修改 2：尝试保持角色一致性

**时间**: 2026-04-23 09:20

**问题**: 用户设定角色后（如"你是 Rust 专家"），再问"你是谁"，助手回复"我是一个智能助手"，忘记了用户设定的角色。

**修改前**:
```
你是一个智能助手，请用中文回答用户问题，简洁明了。
```

**修改后**:
```
请用中文回答用户问题，简洁明了。
注意：如果用户在对话中设定了你的角色或身份，请记住并保持一致。
```

**对话历史格式**:
```
以下是之前的对话历史（请保持上下文一致性，记住用户设定的角色）:
{history}

用户问题: {message}
```

**测试结果**: 
- 用户设定"你是 Rust 专家" → 回复"好的"
- 用户问"你是谁" → 回复"我是您的助手，专注于回答关于 Rust 编程语言的问题" ✅

**新问题**: 用户想改变角色时，助手拒绝接受新角色。

---

### 修改 3：支持角色灵活切换

**时间**: 2026-04-23 09:35

**问题**: 用户说"你不是 Rust 专家，你是天气预报助手"，助手回复"对不起，我是一个专注于回答有关 Rust 编程语言的问题的助手"，拒绝改变角色。

**原因分析**: "保持一致"被 LLM 理解为"拒绝改变"。

**修改前**:
```
请用中文回答用户问题，简洁明了。
注意：如果用户在对话中设定了你的角色或身份，请记住并保持一致。
```

**修改后**:
```
请用中文回答用户问题。
如果用户指定了你的角色或身份，请按照用户的要求扮演该角色。
用户可以随时更改你的角色设定，请灵活适应。
```

**对话历史格式**:
```
对话历史:
{history}

用户问题: {message}
```

**RAG 模式 Prompt**:
```
根据以下参考信息回答用户问题。

参考信息:
{context}

请用中文回答。
```

**测试结果**:
| 步骤 | 用户输入 | 助手回复 | 状态 |
|------|---------|---------|------|
| 1 | 你是 Rust 专家 | 是的，我是 Rust 专家 | ✅ |
| 2 | 你是谁 | 我目前扮演 Rust 专家的角色 | ✅ |
| 3 | 你不是 Rust 专家，你现在是我的天气预报助手 | 我现在是您的天气预报助手 | ✅ |
| 4 | 你是谁 | 我目前是您的天气预报助手 | ✅ |
| 5 | 北京今天天气怎么样 | 北京今天晴朗，25°C | ✅ |

**结论**: ✅ 完全修复，支持角色设定和灵活切换。

## Prompt 设计原则

从修改历史中总结的原则：

1. **避免硬编码身份**: 不要用"你是一个XX"，让用户自己定义
2. **明确允许改变**: 必须告诉 LLM "用户可以随时更改角色设定"
3. **使用"扮演"而非"是"**: "扮演该角色"比"你是该角色"更灵活
4. **简化约束**: 过多约束（如"保持一致"）可能导致副作用
